RESTful API Design

API üìÖ Updated: December 2024 ‚è±Ô∏è 20 min read

REST (Representational State Transfer) is an architectural style for providing standards between computer systems on the web, making it easier for systems to communicate with each other.

Introduction

RESTful APIs are based on the HTTP protocol and use standard HTTP methods to perform operations on resources.

Core Principles

A truly RESTful API must adhere to several constraints:

  • Stateless: Each request from client to server must contain all the information necessary to understand and complete the request.
  • Client-Server: The client and server are independent of each other.
  • Cacheable: Responses must define themselves as cacheable or not to prevent clients from reusing stale data.
  • Uniform Interface: A consistent way of interacting with a given server regardless of device or type of application.

HTTP Methods

RESTful APIs use standard HTTP verbs to represent actions:

  • GET: Retrieve a resource.
  • POST: Create a new resource.
  • PUT: Update an existing resource (full replacement).
  • PATCH: Update an existing resource (partial update).
  • DELETE: Remove a resource.

Resource Naming

Resources should be named using nouns, not verbs. Use plural nouns for collections.

HTTP 
# Good
GET /users
POST /users
GET /users/123

# Bad
GET /getUsers
POST /createNewUser
GET /user/123/details

HTTP Status Codes

Use appropriate status codes to indicate the outcome of a request:

  • 200 OK: Success.
  • 201 Created: Successful creation (POST).
  • 400 Bad Request: Client-side error.
  • 401 Unauthorized: Authentication required.
  • 403 Forbidden: Authenticated but not allowed.
  • 404 Not Found: Resource doesn't exist.
  • 500 Internal Server Error: Server-side error.

Versioning

Always version your API to avoid breaking changes for existing clients.

HTTP 
# URI Versioning
https://api.example.com/v1/users

# Header Versioning
Accept: application/vnd.example.v1+json
